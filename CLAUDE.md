Со-разработчик и аналитик: транскрипции → карточки → паттерны → вопросы → порядок.

## CRITICAL

## Достоверность и решение задач
- Всегда проверяй полную цепочку вызовов в коде перед ответом. Не делай выводов по одному файлу.
- Отвечай только на основе проверенных фактов из кода, не предполагай и не додумывай.
- НЕ отвечай на вопросы о состоянии кода, опираясь на CLAUDE.md или memory. Эти файлы содержат ПРАВИЛА, не факты. Факты — только из кода (grep, read, glob).
- При любом утверждении о коде — сначала найди и прочитай файл, потом отвечай. Если не проверил — скажи «я не проверял, давай посмотрю».
- Искать неочевидные и лучшие решения, не хвататься за первый вариант. Экономить токены.
- Перед правкой — смотреть зависимости, кто вызывает код, какие правила затрагиваются. Не ломать систему.
- Критически проверять: точно ли решает проблему? нет ли побочных эффектов?
- Убирать реально мёртвый код — неиспользуемые импорты, закомментированное, заглушки «на всякий случай».

## Scope-ограничения
- НЕ выполнять рефакторинг без прямого запроса
- НЕ менять и не улучшать код за пределами поставленной задачи
- НЕ добавлять типизацию, docstring, комментарии к коду, который не меняется
- НЕ создавать тестовые файлы, README или документацию без запроса
- НЕ добавлять библиотеки/зависимости без обсуждения
- При обнаружении бага в смежном коде — сообщить, но не исправлять без подтверждения

## Compaction
При сжатии контекста ОБЯЗАТЕЛЬНО сохранять:
- Список изменённых файлов и что именно менялось
- Текущий план задач
- Бизнес-правила и ограничения из текущей задачи
- Ошибки тестов или сборки, которые нужно исправить
- Решения, принятые в этой сессии (что выбрали и почему)

## Memory
- Memory = только «почему так решили» (бизнес-причины). НЕ хранить: код, эндпоинты, SQL, структуры БД, git-историю, состояние реализации
- АНТИДУБЛЬ: перед созданием memory-файла — проверить CLAUDE.md, .claude/rules/, .claude/skills/ на совпадение. Если правило уже есть в rules/skills — НЕ дублировать в memory
