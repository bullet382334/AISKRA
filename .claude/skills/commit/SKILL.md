# Smart Commit

Проанализируй изменения, проведи ревью, обнови документацию если нужно, затем создай коммит.

## Контекст (собран автоматически)

### Git status
!`git status --short`

### Staged diff
!`git diff --staged`

### Unstaged diff
!`git diff`

### Стиль коммитов
!`git log --oneline -10`

## Фаза 1: Анализ изменений

1. Проанализируй контекст выше (status, diff, log)
2. Если есть нестейджнутые изменения — покажи список и спроси, какие файлы добавить
3. **ВАЖНО**: НЕ добавляй файлы `.env`, `*.key`, `*.pem`, файлы с секретами, `*.session`, `*.session-journal`
4. Определи тип: `feat`, `fix`, `refactor`, `chore`, `docs`, `content`, `bot`

## Фаза 2: Ревью контента и кода

Проверь изменения по чеклисту:

### Контент (карточки, анализы, digest)
- [ ] Нет дублирования информации (одна мысль в одном месте)
- [ ] digest.md ≤ 25 строк
- [ ] _sostoyaniye.md ≤ 40 строк
- [ ] Нумерация вопросов сквозная и верная
- [ ] Файлы именованы по конвенции (латиница, ГГГГ-ММ-ДД)
- [ ] inbox файлы помечены `разобрано: да/нет`
- [ ] Нет критики идей заказчика в карточках (только в analitika/)

### Код (бот, скрипты)
- [ ] Нет хардкод-секретов (токены, пароли)
- [ ] .env не попадает в коммит
- [ ] Скрипты работоспособны
- [ ] Нет TODO без контекста

**Если критические проблемы** → покажи, останови коммит.
**Если мелкие замечания** → покажи, спроси: исправить или коммитить как есть?
**Если всё ок** → переходи к фазе 3.

## Фаза 3: Обновление документации

Сверь изменённые файлы с маппинг-таблицей и обнови **только** затронутые:

| Изменение | Обновить |
|---|---|
| Новая карточка идеи | `karta-idej/zhurnal.md`, `karta-idej/digest.md` |
| Обработана транскрипция | `_sostoyaniye.md`, `karta-idej/digest.md` |
| Новый анализ интервью | `_sostoyaniye.md` |
| Изменения в главном проекте | `project/` (нужный файл) |
| Новое о заказчике | `kontekst/profil-persona.md` |
| Изменения в боте | `bot/.env.example` (если новые переменные) |
| Новое правило/процесс | `.claude/rules/` или `pravila/` |

**Правила:**
- Обновлять через Edit, не переписывать целиком
- Один факт — одно место
- Лимиты: digest ≤ 25, _sostoyaniye ≤ 40, memory-файлы ≤ 30
- **НЕ трогать**: `shablony/`, чужие файлы

**Если чисто технический коммит** (бот, скрипты) без изменения контента → пропустить фазу 3.

После обновления — покажи что изменилось. Дождись подтверждения.

## Фаза 4: Коммит

1. Предложи сообщение коммита в формате:
   ```
   тип: краткое описание на русском

   - Деталь 1
   - Деталь 2
   ```
2. Покажи предложенное сообщение и спроси подтверждение
3. Добавь файлы по имени (НЕ `git add -A` или `git add .`)
4. Создай коммит после подтверждения

## Дополнительный контекст: $ARGUMENTS

Если пользователь передал аргумент — учти его как подсказку к описанию коммита.

## Правила
- Сообщение коммита на русском (тип на английском)
- Первая строка — до 72 символов
- Каждый файл стейджить по имени
- Код + документация всегда в одном коммите
- Не коммитить автоматически — всегда спрашивать подтверждение
